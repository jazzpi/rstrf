# v0.2.0

The three big things in this release are

- a major UI rework
- major redesign of how spectrograms are represented, which mean rSTRF now
  properly handles live-recorded (i.e. variable-rate) STRF data
- signal export (`out.dat`)

In addition, there's a bunch of new features, bugfixes and big performance
improvements on large datasets.

Thanks a lot to the people in [`#rstrf`](https://matrix.to/#/#rstrf:matrix.org)
(especially Ben and Jan for being beta-testers and Cees for the explanations of
STRF's design).

## Changes
### UI Rework

The old splittable pane grid is gone. Panes have been replaced by separate
windows. This should hopefully be a bit more flexible, and if you want the old
behavior, you can still have plot & TLE manager windows side-by-side.

Workspaces are also gone, at least for now. Maybe they will make a return in a
future release.

The CLI has changed significantly as well. You can now run `rstrf plot ...` with
an `rfplot`-like interface. For example:
```sh
rstrf plot /path/to/rffft-data/2026-02-19T00\:00\:01_0000{00..99}.bin \
  -c /path/to/tles/bulk_260219_filtered.tle \
  -F /path/to/strf/frequencies.txt  \
  -C 4801 --zmin -38 --fmin 435.2e6 --fmax 435.7e6
```

**NOTE**: you still need to specify all files, not just the prefix up to the
index like in `rfplot -p ...`.

Run `rstrf plot -h` for a full list of arguments.

### Spectrogram representation

Previously rSTRF assumed spectra were recorded at a constant rate. That's fine
for data generated by running `rffft` on IQ files, but when `rffft` is running
on live data, it -- by design -- does not record at a constant rate. rSTRF did
not handle that case, and so aborted while loading the files.

Now, rSTRF handles this correctly. That means it should now load whatever `.bin`
files you throw at it. Unlike `rfplot`, you will see gaps in the plot if there
is a gap in the data. The shader closes very small (1px) gaps though, otherwise
panning through the plot becomes very flickery.

### Signal export

- mark trackpoints with `s` -> click, then find signals with `f`
- mark signals directly with `SHIFT`+`d`
- delete signals with `d` -> click

Once you have all the signals you want, press the *Save* button in the toolbar
to save them to a `.dat` file.

### Other changes

- **Progress indicator** while spectrograms load.
- **`pass-png` subcommand:** batch-generate a PNG for each pass of a given
  satellite (by NORAD ID and transmitter frequency). See [the README](README.md)
  for details.
- **Support multiple TX frequencies per satellite**. Parsed from an STRF
  `frequencies.txt`, or you can temporarily add/delete them from the satellite
  manager.
- **More navigation and view controls:** zoom-to-rectangle (`z`), pan with the
  arrow keys (half-width with `Shift`), `Ctrl`/`Shift`+scroll for
  vertical/horizontal zoom, reset view (`r`), and toggle predictions (`p`).
  Maximum zoom is now bounded by the spectrogram extent.
- **Absolute time & frequency axes** by default. You can toggle between
  relative/absolute axes from the toolbar.
- **Save a screenshot** of the current view.
- **Viridis is now the default colormap** (still configurable in preferences).
- **Windows is now supported** - see the new build instructions in the README.
  Thanks [@ev3nvy](https://github.com/ev3nvy)!
- **Follow STRF site:** rSTRF can now automatically read the observer ground
  site from STRF's `sites.txt` based on your `$ST_SITES_TXT` / `$ST_DATADIR` /
  `$ST_COSPAR` environment variables & the `-C` command-line option. Set "Follow
  STRF site" in Preferences.
- **Prediction status indicator:** a status indicator is shown while Doppler
  pass predictions are being computed.
- `-v`/`-vv` flags to increase log verbosity.
- Significant performance improvements.
- Various bug fixes.

# v0.1.1

This is a patch release that focuses on improvements for STRF workflows that
involve long recordings and large TLE catalogs.

Thanks a lot to the people in
[`#satnogs`](https://matrix.to/#/#satnogs:matrix.org) (and especially
`@cat:nannk.xyz`) for testing this and providing test data!

## Changes

- be more lenient with inconsistent spectrogram slice lengths (these are
  generated by `rffft` running on live data)
- support TLEs with Alpha-5 NORAD IDs
- be more tolerant for partially-invalid TLE catalogs
- support spectrograms that are too large for a single GPU buffer
- various performance optimizations
